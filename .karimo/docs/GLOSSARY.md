# KARIMO Glossary

A comprehensive reference for all KARIMO terminology, concepts, and components.

---

## Core Concepts

### PRD (Product Requirements Document)
Full feature specification created through structured interviews. Contains project context, detailed requirements, acceptance criteria, and decomposed tasks. Stored in `.karimo/prds/{slug}/prd.md`.

### Task
Atomic unit of work representing one logical change (typically affecting 1 file or a small, cohesive set of changes). Tasks are the execution unit assigned to worker agents.

### Wave
Group of independent tasks that execute in parallel. Tasks in Wave 2 cannot start until all Wave 1 tasks are merged. Waves enforce dependency ordering without over-serializing work.

### Brief
Self-contained task instructions generated by the brief-writer agent. Contains all context needed for a worker agent to execute a task without accessing the full PRD. Ensures task portability.

### Worktree
Isolated git workspace that allows multiple branches to be checked out simultaneously. KARIMO uses worktrees via `isolation: worktree` to prevent branch conflicts during parallel task execution.

### DAG (Directed Acyclic Graph)
Visual representation of task dependencies showing which tasks must complete before others can start. Generated during PRD review to validate execution feasibility.

### Complexity
Numeric scale (1-10) indicating task difficulty and scope:
- **1-4**: Simple tasks, uses Sonnet-tier agents (implementer, tester, documenter)
- **5-10**: Complex tasks, uses Opus-tier agents (implementer-opus, tester-opus, documenter-opus)

Determines model selection and agent escalation strategy.

### Loop
Revision cycle in automated code review:
1. Agent submits PR
2. Review provider (Greptile/Code Review) analyzes code
3. If issues found, agent revises
4. Re-review occurs
5. Hard gate after 3 loops (requires human review)

### Slug
URL-friendly identifier for a PRD (e.g., `user-authentication`, `payment-integration`). Used in folder names and command arguments.

### Feature Branch
Long-lived git branch (e.g., `feature/user-auth`) that contains all PRs for a single PRD. Created by `/karimo:run`, merged to main via `/karimo:merge`.

### Finding
Issue or concern discovered during pre-execution review (brief-reviewer agent). Critical findings must be corrected before execution begins. Stored in `findings.md`.

## Agents (22 Total)

### Coordination Agents (16)

#### karimo-interviewer
Conducts structured PRD interviews using a question-driven workflow. Gathers requirements, acceptance criteria, technical constraints, and user stories.

#### karimo-investigator
Scans codebase to detect runtime, framework, testing patterns, and architectural conventions. Populates `.karimo/config.yaml` during initial setup.

#### karimo-reviewer
Validates completed PRD for quality, generates dependency graph (DAG), assigns complexity scores, and organizes tasks into waves.

#### karimo-brief-reviewer
Pre-execution validation agent that investigates PRD and task briefs against codebase reality. Produces findings document before execution starts.

#### karimo-brief-corrector
Applies fixes to task briefs and PRD based on review findings. Modifies `prd.md`, `tasks.yaml`, and briefs as needed to resolve critical issues.

#### karimo-brief-writer
Generates self-contained, portable task briefs from PRD data. Spawned during Phase 1 of `/karimo:run` for each task.

#### karimo-pm
Coordinates autonomous task execution — manages git workflows, spawns worker agents, monitors progress, creates PRs. **Never writes code.** Acts as orchestrator only.

#### karimo-review-architect
Code-level integration and merge quality agent. Validates task PRs, resolves merge conflicts, performs feature reconciliation when PM Agent needs validation.

#### karimo-feedback-auditor
Investigates complex feedback issues requiring evidence gathering. Produces findings embedded in feedback documents for system improvement.

### Worker Agents (6)

#### karimo-implementer (Sonnet)
Executes coding tasks (complexity 1-4) from KARIMO PRDs. Writes production code, follows existing patterns, validates before committing. For straightforward implementation and bug fixes.

#### karimo-implementer-opus (Opus)
Executes complex coding tasks (complexity 5+) with deep reasoning and multi-file coordination. For high-complexity implementation and architectural changes.

#### karimo-tester (Sonnet)
Writes and maintains tests (complexity 1-4). Creates unit tests, integration tests, validates coverage. For straightforward test-specific tasks.

#### karimo-tester-opus (Opus)
Writes tests for complex KARIMO tasks (complexity 5+). Creates comprehensive test suites with deep coverage analysis.

#### karimo-documenter (Sonnet)
Creates and updates documentation (complexity 1-4). Writes README updates, API docs, component docs, inline documentation.

#### karimo-documenter-opus (Opus)
Creates complex documentation (complexity 5+). Writes comprehensive API docs and architectural documentation requiring deep technical analysis.

## Status States

Tasks and PRDs progress through defined lifecycle states:

### PRD States
- **draft**: Interview in progress, not yet approved
- **ready**: PRD approved, ready for execution
- **executing**: Tasks currently running
- **completed**: All tasks merged successfully
- **stalled**: No progress in 24+ hours

### Task States
- **queued**: Task waiting for its wave to start
- **running**: Task currently executing (agent spawned)
- **in-review**: PR created, awaiting code review
- **needs-revision**: Review found issues, agent must fix
- **completed**: PR merged to feature branch
- **failed**: Task aborted after 3 revision loops
- **stalled**: No progress in 24 hours (requires investigation)

├── tasks.yaml          # Task definitions with dependencies
├── status.json         # Execution state tracking
├── findings.md         # Pre-execution review findings
├── dependencies.json   # Task dependency graph

## Command Categories

### Core Workflow
- **`/karimo:plan`** — Create new PRD through structured interview
- **`/karimo:run --prd {slug}`** — Execute PRD tasks (feature branch workflow)
- **`/karimo:dashboard`** — Check execution progress (all PRDs)
- **`/karimo:dashboard --prd {slug}`** — Detailed status for specific PRD
- **`/karimo:merge --prd {slug}`** — Create final PR from feature branch to main

### Configuration
- **`/karimo:configure`** — Project setup (basic mode, auto-detection)
- **`/karimo:configure --advanced`** — Full 9-step configuration
- **`/karimo:configure --preview`** — Preview config without saving
- **`/karimo:configure --validate`** — Validate existing config against codebase

### Management
- **`/karimo:feedback`** — Capture learnings (auto-detects simple vs. complex feedback)

### Diagnostics
- **`/karimo:doctor`** — Health check with 7 diagnostic categories
- **`/karimo:doctor --test`** — Installation smoke test (end-to-end verification)
- **`/karimo:dashboard`** — Comprehensive metrics and analytics view

### Maintenance
- **`/karimo:update`** — Update to latest KARIMO version from GitHub
- **`/karimo:configure --cd`** — Configure preview deployment providers (Vercel, Netlify, etc.)

## Execution Models

### Feature Branch Model (Recommended)
**Command:** `/karimo:run --prd {slug}`

**Workflow:**
1. Creates feature branch `feature/{prd-slug}`
2. Tasks create PRs targeting feature branch
3. Wave-based parallel execution (Wave 2 waits for Wave 1)
4. Each PR reviewed independently
5. Merge PRs to feature branch as they complete
6. Final PR to main via `/karimo:merge --prd {slug}`

**Benefits:**
- Complete isolation from main branch
- Parallel task execution within waves
- Independent PR review and merging
- Rollback entire feature if needed

### Direct-to-Main Model (Legacy, v4.0)
**Command:** `/karimo:run --prd {slug}`

**Workflow:**
1. Tasks create PRs directly to main
2. Wave-based execution still applies
3. Labels used for tracking (`prd:slug`, `wave:1`)
4. No final merge step needed

**Use cases:**
- Solo developers
- Hotfixes
- Small changes (1-2 tasks)
- Projects not using feature branches

## Wave Execution Rules

### Wave Dependencies
- **Wave 1** → Always runs first (no dependencies)
- **Wave 2** → Runs after all Wave 1 tasks merged
- **Wave N** → Runs after all Wave N-1 tasks merged

### Parallelism
- Tasks within a wave execute in parallel
- Each task gets isolated worktree
- PRs created simultaneously
- Merges happen independently as reviews pass

### Finding Propagation
When a task in Wave 1 discovers new information:
1. PM Agent pauses Wave 2 execution
2. Briefs for dependent tasks updated
3. Wave 2 resumes with corrected context

## Configuration Modes

### Basic Mode (Default)
**Questions:** 3
**Time:** ~5 minutes
**Questions:**
1. Confirm detected runtime/framework?
2. Which files should agents never touch?
3. Enable automated review?

**Questions:** 9

**Invoke:** `/karimo:configure --advanced`

### Auto Mode (CI/Testing)
**Questions:** 0
**Time:** < 1 minute
**Behavior:** Accepts all defaults, no interaction
**Invoke:** `/karimo:configure --auto`

## Boundaries

### `never_touch`
Glob patterns for files agents must NEVER modify:
```yaml
boundaries:
  never_touch:
    - "node_modules/**"
    - ".env*"
    - "*.lock"
    - "db/migrate/**"
```

Agents will refuse to edit these files and report errors if tasks target them.

### `require_review`
Glob patterns for files requiring human approval before modification:
```yaml
boundaries:
  require_review:
    - "src/security/**"
    - "config/auth.ts"
    - "package.json"
```

Agents can edit these files but PRs will be flagged for mandatory human review.

## Key Metrics

### Execution Metrics
- **Total tasks:** Count across all PRDs
- **Completed tasks:** Successfully merged
- **Failed tasks:** Aborted after 3 loops
- **Average complexity:** Mean across all tasks
- **Wave count:** Number of dependency waves

### Review Metrics
- **Review pass rate:** First-attempt success percentage
- **Average loops per task:** Revision cycles needed
- **Model escalation rate:** Sonnet → Opus frequency
- **Hard gate hit rate:** Tasks requiring human review

### Performance Metrics
- **Time to first PR:** PRD approval → first PR created
- **Average task duration:** Start → merge time
- **Wave parallelism:** Average tasks per wave
- **Merge latency:** Review pass → merge time
